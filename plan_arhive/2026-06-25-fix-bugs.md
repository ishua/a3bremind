> План по исправлению критических и важных проблем, выявленных в ходе ревью кода a3bremindbot.

Без задачи (инициатива по результатам ревью)

## Контекст

В ходе ревью проекта a3bremindbot выявлено 4 критических и 7 важных проблем. Проект — Telegram-бот для напоминаний на Go, использует SQLite (modernc.org/sqlite), чистая архитектура (store → domain → bot). Тесты есть, используют `:memory:` SQLite с `MaxOpenConns(1)`.

Пользователь выбрал скоуп: критичные + важные, с тестами для каждого исправления.

## Цель

Устранить все 11 проблем из ревью, добавив тесты на каждое исправление. Пройти `make test-all` (`go test ./... -v -race -count=1`) без ошибок после всех изменений.

## Решения и договорённости

- **Транзакции через интерфейс Querier (подход А)**: в store определяется интерфейс `Querier` с методами `Exec`, `Query`, `QueryRow`. Все store-функции переходят с `*sql.DB` на `Querier`. `*sql.DB` и `*sql.Tx` оба реализуют этот интерфейс. NextInstance и все связанные операции выполняются в одной транзакции.
- **Store остаётся low-level**: транзакция начинается и коммитится в вызывающем коде (bot/domain), store только принимает `Querier`.
- **Context.Context не добавляется в этом раунде**: это слишком инвазивное изменение (все функции, все тесты), которое не относится к багам. Стоит сделать отдельной итерацией.
- **GetInstanceByMessageID через json_each**: альтернатива — отдельная таблица `instance_message_ids`, но это требует миграции данных и меняет схему. json_each — минимальное изменение.
- **Тесты обязательны**: для каждого исправления тест (или группа тестов), воспроизводящий баг и проверяющий исправление.
- **Порядок фаз важен**: Фаза 1 должна быть первой (безопасность), Фаза 2 — второй (меняет API), Фазы 3-4 независимы и могут идти в любом порядке после 1-2.

## Отложено

- `context.Context` для store-функций — отдельная итерация
- `handleConfirmDoneTime` не проверяет paused — intentional по задумке, оставляем
- Ответ на неизвестный текст (подсказка) — низкий приоритет
- `PendingConfirm` только в памяти — low priority, документировано
